# Claude Skills Framework Standards

> **This is the constitution.** Every skill — new, modified, or refactored — must comply.
> Automated enforcement runs via `setup.sh`. Manual review catches what automation cannot.

## 1. Step 0-N Protocol

Every SKILL.md body follows this structure:

| Step | Purpose | Required? |
|------|---------|-----------|
| **Step 0: Usage Logging** | `echo "$(date -u +%Y-%m-%dT%H:%M:%SZ) <skill-name>" >> ~/.claude/skill-usage.log` | **Always** |
| **Step 1** | Gate check, input validation, preconditions | When applicable |
| **Step 2** | Core logic — the actual work | **Always** |
| **Step N** | Output, report, or context wipe | **Always** |

Planning skills (`/plan-review`, `/project-kickoff`) end with **The Context Wipe**: instruct the user to `/clear` before execution. This prevents brainstorming context from polluting the execution phase.

## 2. Portability

**Absolute prohibition of hardcoded paths.**

| Context | Use | Example |
|---------|-----|---------|
| Referencing files *within* the skill's own directory | `${CLAUDE_SKILL_DIR}` | `cat ${CLAUDE_SKILL_DIR}/references/guide.md` |
| Referencing files *outside* the skill (other projects) | `$PROJECTS_DIR` | `cd $PROJECTS_DIR/my-project` |
| Runtime environment variables | Source first | `source ~/.claude/skills-env.sh` |

Never write `/root/projects/...` in a SKILL.md. The `${CLAUDE_SKILL_DIR}` variable is resolved by the agent runtime to the skill's installed directory. `$PROJECTS_DIR` is set by `skills-env.sh` (generated by `setup.sh`).

## 3. Progressive Disclosure

SKILL.md is a lightweight router, not a monolith. **Hard limit: 500 lines.**

| Content type | Location | Loaded when |
|-------------|----------|-------------|
| Orchestration logic | `SKILL.md` | Always (injected into context) |
| Templates, scaffolds | `assets/` | On demand via `cat ${CLAUDE_SKILL_DIR}/assets/...` |
| Reference docs, checklists | `references/` | On demand via `cat ${CLAUDE_SKILL_DIR}/references/...` |
| Executable scripts | `scripts/` | On demand via `bash ${CLAUDE_SKILL_DIR}/scripts/...` |

If a section of SKILL.md is only needed conditionally (e.g., "if the user asks for API docs"), extract it to a reference file and load it with a conditional `cat`. Every token in SKILL.md is injected into context on every invocation — make each one earn its place.

## 4. Domain Isolation

| Scope | Contains | Example |
|-------|----------|---------|
| `user/` | Stack-agnostic skills usable across all projects | `/architect`, `/code-review`, `/sec-audit` |
| `<project>/` | Project-scoped skills with domain-specific logic | e.g., `/cx-architect`, `/hl-deploy` |

Example project scopes (for illustration):

| Scope | Contains |
|-------|----------|
| `my-webapp/` | Web application-specific logic (framework, ORM, auth patterns) |
| `infra/` | Infrastructure operations (orchestration, provisioning, monitoring) |
| `my-cli/` | CLI tool-specific workflows |

**Rule:** A `user/` skill must never reference project-specific frameworks, APIs, or domain concepts. If domain logic is needed, create a project-scoped variant (e.g., `/decision-critic` → `/my-project-decision-critic`).

## 5. Safety Gates

### Dry-run defaults

Skills that mutate infrastructure or deploy code must default to `--dry-run` mode. The SKILL.md must show what *would* happen and require explicit user confirmation before executing. Skills with side effects set `disable-model-invocation: true` in frontmatter to prevent autonomous triggering.

Examples of mutator skills: `/init-repo`, `/wrap-up`, and any project-scoped deploy or migration skills.

### CHANGELOG gate

Every skill modification (create, improve, archive, restore) must update `CHANGELOG.md` under `[Unreleased]` before committing. Enforced by `/code-review` and `/skill-manager`.

### Git safety

- Stage files explicitly — never `git add -A` or `git add .`
- Never commit `.env`, `*.pem`, `*.key`, `secrets/`, `credentials*`
- Include `Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>` in AI-generated commits

## 6. Frontmatter (agentskills.io spec)

| Field | Required | Rules |
|-------|----------|-------|
| `name` | Yes | Lowercase + hyphens, no consecutive hyphens, max 64 chars, must match directory name |
| `description` | Yes | Max 1024 chars. State what it does AND when to trigger it. Slightly "pushy" to combat under-triggering |
| `argument-hint` | Optional | Short hint shown to user (e.g., `"[project name]"`) |
| `allowed-tools` | Optional | Restrict which Claude Code tools the skill can use |
| `disable-model-invocation` | Optional | `true` for mutator skills with side effects |

## 7. Writing Style

- **Positive framing** — "Do X because [reason]" instead of "NEVER do X"
- **Imperative mood** — "Read the file" not "You should read the file"
- **Explain the why** — models follow instructions better with reasoning
- **One instruction, one place** — no redundancy across sections
